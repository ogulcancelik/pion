Scheduled jobs currently record runner/runtime failures in cron state (`lastStatus`, `lastError`) but do not notify the target Telegram chat when a scheduled run fails.

This is too invisible for background work. Users expect scheduled jobs to either deliver a result or say they failed.

## Problem

Cron `agent` jobs reuse the normal runner/model/tool stack, so they can fail for all the same reasons as foreground runs:
- auth/config errors
- provider outages / network failures
- quota / rate limit
- model lookup failures
- runtime/tool exceptions
- partial-output-then-fail cases

Foreground chat turns already have user-facing fallback behavior through Runner + daemon flow.
Cron runs currently only persist failure state internally.

## Goal

When a scheduled job fails, send a concise failure notice to the job's target chat, using safe user-facing error language rather than raw stack traces or provider junk.

## Important constraint

Do not solve this as an ad hoc `catch -> send error.message` in the scheduler.
The right shape is probably shared runtime-error classification reused by both:
- foreground daemon turns
- scheduled cron runs

## Things to investigate

- Which runtime/provider errors should map to user-safe messages?
- Can existing `Runner` error classification be reused directly?
- Should cron have a thin wrapper like `Scheduled job "<name>" failed: ...` around shared classified messages?
- How should partial-output-then-fail runs behave?
- Should `Runner.process()` return richer structured failure info instead of only throwing?
- Should some failures remain internal-only, or should all terminal cron failures notify chat?

## Desired outcome

- scheduled job failures are visible in the target chat
- wording is concise and user-safe
- foreground and cron error surfacing stay consistent
- raw stack traces / auth details are not leaked to users
- cron state/logging still records detailed internal failure info

Implemented.

What changed:
- Scheduled `agent` jobs no longer send every intermediate `onTextBlock` to Telegram.
- Cron delivery now waits for the run to complete and sends only the final assistant text block to the target chat.
- Chat-context append now mirrors what the user actually received, so hidden intermediate cron narration is no longer written into the user-visible session history.
- Added shared `getUserFacingErrorMessage()` in `src/core/runner.ts` and reused it from both foreground daemon error handling and cron scheduled-job failure notices.
- Scheduled job failures now send one concise user-facing Telegram failure message to the target chat instead of staying internal-only.
- Failure paths now write cron output artifacts with prompt, captured response blocks, and internal error details, while still keeping the chat-facing text safe.
